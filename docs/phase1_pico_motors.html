1. Overview

Phase 1 covers everything that runs on the Pi Pico. The Pico is the hardware interface layer — it talks directly to motors and encoders, and communicates with the RB3 over USB serial. The RB3 handles all computation; the Pico only needs to execute commands and report measurements.

Responsibilities of the Pico:

  • Count encoder pulses from both wheels using hardware interrupts
  • Drive motors via a dual-PWM motor driver
  • Receive velocity commands from the RB3 (v_lin, v_ang)
  • Broadcast encoder tick counts to the RB3 at 50 Hz
  • Stop motors automatically if no command is received within 500 ms (watchdog)

Files in this phase:

  • robot_deploy/pico/config.py — all constants
  • robot_deploy/pico/main.py — firmware

2. Hardware setup

Assumed motor driver

A dual-PWM H-bridge driver (e.g. TB6612, DRV8833, L9110S). Each motor has two PWM pins: one for forward, one for reverse. Setting P1 high and P2 low drives forward; P1 low and P2 high drives reverse. Both low = coast. Both high = brake.

This is different from an L298N (which uses one PWM + two direction pins). If your driver uses ENA/IN1/IN2, the Motor class needs to be rewritten — see section 12.

3. Communication protocol

The Pico communicates with the RB3 over USB serial (115200 baud). Both directions use newline-delimited JSON — one JSON object per line.

Pico → RB3 (50 Hz)

{"tick_l": 12, "tick_r": 11, "tpr": 990, "ts": 34.5}
FieldTypeMeaning
tick_lintLeft encoder ticks since last packet (can be negative)
tick_rintRight encoder ticks since last packet
tprintTicks per wheel revolution (990) — lets RB3 convert to speed
tsfloatMilliseconds since Pico boot, divided by 1000

RB3 → Pico (20 Hz)

{"v_lin": 0.25, "v_ang": -0.1}
FieldTypeMeaning
v_linfloatForward velocity in m/s (negative = reverse)
v_angfloatAngular velocity in rad/s (positive = turn left)

Motor test command (from viser panel)

{"d_l": 50.0, "d_r": 50.0}

Sets motor duties directly in percent [-100, 100]. Used for diagnosis. Detected by the presence of d_l or d_r keys instead of v_lin/v_ang.

4. Encoder

A quadrature encoder produces two square waves (A and B) offset by 90°. By reading A on a rising edge and checking the level of B, you can determine direction. 

A: ‾|_|‾|_|‾|_|
B:  ‾|_|‾|_|‾|_     ← B lags A → forward

Rule: if A rises and B is high → forward (+1). If A rises and B is low → reverse (−1).

Class: Encoder

class Encoder:
    def __init__(self, pin_a, pin_b):
        self._ticks = 0
        self._a = Pin(pin_a, Pin.IN, Pin.PULL_UP)
        self._b = Pin(pin_b, Pin.IN, Pin.PULL_UP)
        self._a.irq(trigger=Pin.IRQ_RISING, handler=self._isr)

    def _isr(self, _pin):
        self._ticks += 1 if self._b.value() else -1

    def pop(self):
        t = self._ticks
        self._ticks = 0
        return t           # returns ticks since last call, resets counter
Only channel A generates interrupts. Channel B is read as a level — this avoids double-counting that occurs if both channels trigger interrupts. CPR = 22 means 22 rising edges per encoder shaft revolution.

Converting ticks to wheel speed (done on the RB3 side)

wheel_circumference = 2 * pi * 0.0335       # 0.2105 m
wheel_speed = (ticks / 990) * wheel_circumference / dt     # m/s

5. Motor driver

Each motor is controlled by two PWM pins. Duty cycle is 0–65535 (MicroPython duty_u16 range). Frequency is set to 20 kHz to keep the PWM inaudible and reduce motor heating.

StateP1P2
Forward at X%X% duty0
Reverse at X%0X% duty
Coast (stop)00
Brake100%100%

Class: Motor

class Motor:
    def __init__(self, p1_pin, p2_pin):
        self._p1 = PWM(Pin(p1_pin)); self._p1.freq(20000)
        self._p2 = PWM(Pin(p2_pin)); self._p2.freq(20000)

    def set_duty(self, duty):          # duty in [-100, 100]
        duty = max(-100.0, min(100.0, duty))
        u16  = int(abs(duty) / 100.0 * 65535)
        if duty >= 0:
            self._p1.duty_u16(u16); self._p2.duty_u16(0)
        else:
            self._p1.duty_u16(0);   self._p2.duty_u16(u16)

    def stop(self):
        self._p1.duty_u16(0); self._p2.duty_u16(0)

6. Drive train

The drive train converts a body velocity command into individual wheel duties. This is standard differential drive kinematics.

Differential drive equations

v_left  = v_lin - v_ang * (WHEEL_BASE / 2)
v_right = v_lin + v_ang * (WHEEL_BASE / 2)

A positive v_ang means turning left (counter-clockwise when viewed from above): the left wheel slows and the right wheel speeds up.

Normalisation

If either wheel speed exceeds max_v, both are scaled down proportionally so the robot turns at the correct ratio but within motor limits. 

peak = max(|v_left|, |v_right|, max_v)
duty_left  = (v_left  / peak) * 100
duty_right = (v_right / peak) * 100

Invert flags

If a motor is physically wired in reverse, set INVERT_LEFT = True or INVERT_RIGHT = True in config.py. The duty is negated before being sent to that motor. Symptom of needing this: robot spins in circles when given a straight command.

7. Serial I/O

On MicroPython, the USB serial port is exposed as sys.stdin / sys.stdout. Reading must be non-blocking so the encoder interrupts and motor watchdog are not delayed.

Non-blocking read

import uselect as select

_rx_buf = ""

def _try_readline():
    r, _, _ = select.select([sys.stdin], [], [], 0)   # 0 = non-blocking
    if r:
        ch = sys.stdin.read(1)
        if ch in ("\n", "\r"):
            line    = _rx_buf.strip()
            _rx_buf = ""
            return line if line else None
        _rx_buf += ch
    return None
Use uselect, not select — MicroPython's module is named uselect. Using select will cause an ImportError on some firmware versions.

Send

def _send(data):
    sys.stdout.write(ujson.dumps(data) + "\n")

8. Main loop

The main loop runs at roughly 2 kHz (sleep_us(500)). It does three things each iteration:

  1. Try to read one character from USB and parse a complete command
  2. Check the watchdog — stop motors if no command in 500 ms
  3. Every 20 ms, send an encoder snapshot to the RB3
last_broadcast = time.ticks_ms()
last_cmd       = time.ticks_ms()

while True:
    now = time.ticks_ms()

    line = _try_readline()
    if line:
        pkt = ujson.loads(line)
        if "d_l" in pkt or "d_r" in pkt:
            drive.set_individual(pkt["d_l"], pkt["d_r"])   # motor test
        else:
            drive.set_velocity(pkt["v_lin"], pkt["v_ang"])
        last_cmd = now

    # Watchdog
    if time.ticks_diff(now, last_cmd) > 500:
        drive.stop()

    # Broadcast at 50 Hz
    if time.ticks_diff(now, last_broadcast) >= 20:
        _send({"tick_l": enc_l.pop(),
               "tick_r": enc_r.pop(),
               "tpr": 990, "ts": now / 1000.0})
        last_broadcast = now

    time.sleep_us(500)
Use time.ticks_diff() instead of subtraction for time comparisons in MicroPython. The tick counter wraps around and subtraction gives wrong results at the wrap point.

10. Testing checklist

  1. Flash MicroPython
    Hold BOOTSEL, plug in USB, drag .uf2 to RPI-RP2 drive.
  2. Flash scripts 
    mpremote cp config.py :config.py
mpremote cp main.py   :main.py
mpremote ls                      # confirm both files exist
  3. Check for startup errors 
    mpremote repl
# press Ctrl-D to soft reset and see output
    Expected: no traceback, silent (waiting for commands).
  4. Confirm USB serial on RB3 
    lsusb                        # look for 2E8A:0005
ls /dev/ttyACM*              # should show ttyACM0
  5. Stream live odometry from RB3 
    python3 test_hardware.py
    Spin each wheel by hand — the corresponding tick column should change.
  6. Motor test via viser
    Run python3 main.py --no-nav, open the browser, go to Motor Test panel. Set Left = 40, Right = 0, click Run. Only the left wheel should spin forward. Repeat for right. Adjust INVERT_LEFT/RIGHT as needed.
  7. Straight line test
    In viser, switch to Manual Drive. Set Forward = 0.2, Turn = 0. The robot should drive straight. If it curves, measure WHEEL_BASE more carefully or trim MAX_RPM.
Phase 1 is complete when: both wheels spin in the correct direction, encoder ticks count up when moving forward, and the robot drives straight at a constant forward command.

12. Rewrite guide

If you want to rewrite the Pico firmware from scratch, implement these four components in order. Each one is independent.

Component 1 — Encoder counter

Requirements:

  • Two instances (left, right)
  • Hardware interrupt on pin A, rising edge only
  • Read pin B inside the interrupt to determine direction
  • A pop() method that returns the count and resets it atomically

Component 2 — Motor driver

Requirements:

  • Two instances (left, right)
  • PWM on two pins at 20 kHz
  • A set_duty(duty) method accepting [-100, 100]
  • A stop() method that sets both pins to 0
  • If your driver uses ENA + IN1 + IN2 instead: one PWM pin (ENA) and two digital pins (IN1, IN2)

Component 3 — Drive train

Requirements:

  • Takes v_lin (m/s) and v_ang (rad/s)
  • Converts to v_left and v_right using differential drive equations
  • Converts speeds to duties using MAX_RPM and WHEEL_RADIUS
  • Normalises if either duty exceeds 100%
  • Applies invert flags

Component 4 — Main loop

Requirements:

  • Non-blocking serial read — must not stall the loop
  • Parse JSON command, call drive train or motor test as appropriate
  • Reset watchdog timer on every received command
  • Stop motors if watchdog expires (500 ms default)
  • Send encoder packet every 20 ms
  • Sleep 500 µs per iteration to yield time to interrupts

Minimum viable packet format

You are free to change the JSON field names or switch to a binary protocol (e.g. struct-packed bytes), but the RB3 side (hardware.py) must be updated to match. The only constraint is that each message ends with a newline and that tpr is always sent so the RB3 knows how to convert ticks to speed.
